# Lisp / Prolog / 提案方式の比較

更新日: 2026-03-28

## 結論

この方式が `Lisp` や `Prolog` に見えるのは自然である。

理由:

- Lisp 的な木構造と正規化表現を持つ
- Prolog 的な関係、束縛、未解決状態を持つ

ただし、本質はどちらでもない。

この方式は、
`実行言語`
でも
`純論理推論系`
でもなく、
`共通表現レイヤー`
として設計されている。

## 比較表

| 観点 | Lisp | Prolog | 提案方式 |
| --- | --- | --- | --- |
| 中心目的 | 汎用プログラミング、式評価 | 論理関係、探索、証明 | 共通IR、知識接続、低コスト言語処理 |
| 基本データ形 | リスト、S式、木 | 事実、規則、項、関係 | typed slot graph + optional S-expression serialization |
| 主な操作 | 評価、関数適用、再帰 | unification、backtracking、query | 構造割当、束縛、制約、参照、再表現 |
| 曖昧性の扱い | 実装次第、標準では弱い | 複数解を返せる | `choice / conf / unresolved` を第一級に持つ |
| 実行性 | 強い | 強い | コア表現は非TC、実行は外部導出部へ分離 |
| 知識の置き場所 | コードやデータとして内包 | fact/rule として記述 | 外部知識部に分離し runtime で接続 |
| 自然言語への適性 | 低い。直接は不向き | 低い。直接は不向き | 自然言語、数式、regex、Markdown、コードを跨ぐ |
| 潜在表現 | 基本的に持たない | 基本的に持たない | hard layer + soft layer を持つ |
| 文書混在構造 | 標準では弱い | 標準では弱い | Markdown、自然言語、数式、コード混在を想定 |
| 主役のモデル | 実行系 | 探索系 | 小型Transformer + parser + external knowledge の分業系 |

## Lisp に近い点

Lisp に近いのは主に次である。

- 木構造を中心に持つ
- prefix で正規化しやすい
- S-expression で直列化しやすい
- 再帰構造を自然に表せる

つまり、
`見た目の表現形式`
としてはかなり近い。

ただし違うのは、こちらの主目的が
`式の評価`
ではなく
`意味の保存と共有`
である点だ。

この方式では、S-expression は本質ではなく、
あくまで canonical serialization の候補にすぎない。

## Prolog に近い点

Prolog に近いのは主に次である。

- 関係中心である
- 束縛が重要である
- 一部が未解決でも処理を続けられる
- query / fact / rule の感覚がある

つまり、
`関係と束縛`
の考え方はかなり近い。

ただし違うのは、こちらが
`論理探索`
を中心にしていない点である。

本方式では、

- backtracking を万能原理にしない
- すべてを Horn clause に落とさない
- soft layer による連続表現を持つ
- parser と小型Transformerで曖昧入力を扱う

という違いがある。

## いちばん大きい違い

いちばん大きい違いは、提案方式が
`表現と実行を分離する`
ことにある。

Lisp も Prolog も、基本的には

- 表現
- 知識
- 実行

がかなり近接している。

一方、提案方式では、

- 表現部
- 知識部
- 導出部

を明示的に分ける。

したがって、

- IR は理解共有のための層
- 知識は外部KBの層
- 実行は導出部の層

となる。

ここが単なる Lisp 風記法や Prolog 風記法との差である。

## Soft Layer があることの意味

古典的な Lisp や Prolog と決定的に違うのは、
この方式が `soft layer` を持つことだ。

つまり、

- 明示的な slot / relation / statement
- それに付随する潜在表現

を同時に持つ。

これにより、

- 微妙な語義差
- 用法のにじみ
- 談話的ニュアンス
- 類似性
- 未知語の暫定的意味

を保持できる。

この点で、提案方式は
`古典記号系の復活`
ではなく、
`古典記号系を latent representation と外部知識の上で再構成したもの`
である。

## 表現として似るのは問題か

問題ではない。

むしろ良い。

理由:

- 木や関係を明示できる
- 検証しやすい
- 共有参照を入れやすい
- 正規化しやすい

からである。

重要なのは、
`Lisp っぽい見た目`
や
`Prolog っぽい関係性`
を持つこと自体ではなく、
それを
`現代的な知識分離と潜在表現の中で使う`
ことである。

## どう位置づけるのがよいか

最も自然な位置づけは次である。

`Lisp 的な構造表現`  
`Prolog 的な関係・束縛`  
`Transformer 的な潜在空間`  
`RAG 的な外部知識接続`

を分業させた language model substrate である。

この言い方なら、

- 古典系との連続性
- 現代系との差分

の両方が見えやすい。

## 一文での整理

この方式は、Lisp や Prolog に似た表現形式を持ち得るが、本質はそれらの再実装ではなく、記号表現、潜在表現、外部知識を分離統合した共通表現基盤である。
