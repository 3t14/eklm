# 語彙抽象化ポリシー

更新日: 2026-03-28

## 目的

このポリシーの目的は、共通表現の学習コストと推論コストを下げるために、

- 何を固定語彙として扱うか
- 何を抽象化するか
- 未知語をどう扱うか

を明確にすることである。

中心的な考え方は、

`理解の初期段階では、構造的に重要な記号だけを強く扱い、それ以外の内容依存要素はまず抽象化する`

である。

## 基本方針

### 1. 構造記号を優先する

最初にモデル化すべきなのは、世界知識を多く持つ内容語や分野依存記号ではなく、構造を支える高頻度要素である。

優先対象:

- 助詞
- 助動詞
- 接続詞
- 指示詞の一部
- 句読点
- 括弧、引用符、記号
- 数量表現
- 単位表現
- 否定、疑問、条件、時制を示す表現
- 数式演算子
- 関数適用や添字のような数式構造
- 正規表現の構造演算
- パターン境界や文字種クラス

これらは表現の骨格を作るため、固定語彙として持つ価値が高い。

### 2. 内容依存要素は最初から完全理解しない

自然言語の内容語、数式の記号名、正規表現の具体パターンの多くは、初期段階では詳細に解釈しなくてよい。

これらはまず抽象化する。

### 3. 抽象化は完全に潰しすぎない

すべてを単なる `UNK` にすると、構造と意味の橋渡しに必要な情報まで失う。

したがって、未知語には最低限の型や役割を残す。

## 固定語彙に入れるもの

最小セットの候補:

- `は`, `が`, `を`, `に`, `で`, `と`, `へ`, `から`, `まで`, `より`
- `の`, `も`, `しか`, `だけ`, `など`
- `た`, `ている`, `ない`, `れる`, `られる`, `たい`
- `か`, `な`, `よ`, `ね`
- `そして`, `しかし`, `また`, `もし`, `なら`
- `、`, `。`, `？`, `！`, `（`, `）`, `「`, `」`
- `+`, `-`, `*`, `/`, `=`, `<`, `>`
- 数字と数量クラス
- 単位記号
- `sin`, `cos`, `log`, `exp`, `sum`, `prod`
- `(`, `)`, `[`, `]`, `{`, `}`
- regex の `|`, `*`, `+`, `?`, `.`, `^`, `$`
- 文字種クラスや繰り返しの構造タグ

これらは文法・制約・構文境界・論理関係・パターン境界を示すので、内容依存要素より優先して扱う。

## 抽象化対象

抽象化の主対象:

- 固有名詞
- 地名
- 組織名
- 製品名
- 専門用語
- 低頻度一般名詞
- 未知の複合語
- 長いカタカナ語
- 数式中の任意変数名
- 分野依存の関数記号
- 文脈依存の定数記号
- 行列、テンソル、添字付き記号
- 正規表現中の具体的なリテラル列
- ドメイン固有のパターン名

例:

- `東京大学` -> `PROPN_SLOT_1`
- `ChatGPT` -> `TERM_SLOT_1`
- `超伝導量子干渉計` -> `TERM_SLOT_2`
- `\alpha_i` -> `COEFF_SLOT_1`
- `f_{\phi}` -> `FUNC_SLOT_1`
- `x_i` -> `VAR_SLOT_1`
- `[A-Z]{3}[0-9]{4}` -> `PATTERN_SLOT_1`

## 抽象化レベル

未知語は1段階ではなく、多段階で抽象化する。

### レベル0: そのまま保持

高頻度文法語や記号は、そのまま保持する。

### レベル1: 品詞型付き未知語

語の細かい意味は捨てるが、最低限のカテゴリは保持する。

例:

- `NOUN_UNK`
- `VERB_UNK`
- `ADJ_UNK`
- `ADV_UNK`
- `PROPN_UNK`
- `NUM_UNK`
- `UNIT_UNK`
- `VAR_UNK`
- `FUNC_UNK`
- `CONST_UNK`
- `PATTERN_UNK`

### レベル2: スロット化

同一文書や同一文内で同一対象を追跡できるよう、匿名スロットを割り振る。

例:

- `ENTITY_SLOT_1`
- `ENTITY_SLOT_2`
- `TERM_SLOT_1`
- `TERM_SLOT_2`
- `VAR_SLOT_1`
- `FUNC_SLOT_1`
- `CONST_SLOT_1`
- `PATTERN_SLOT_1`

### レベル3: 型付きスロット

さらに役割や種類を残す。

例:

- `PERSON_SLOT_1`
- `ORG_SLOT_1`
- `LOC_SLOT_1`
- `PRODUCT_SLOT_1`
- `FORMULA_SLOT_1`
- `SCALAR_SLOT_1`
- `VECTOR_SLOT_1`
- `MATRIX_SLOT_1`
- `REGEX_SLOT_1`

この段階まで残せば、未知要素でもかなり多くの構造処理が可能になる。

## 具体例

### 例1

原文:

`太郎は新型量子加速器を買った。`

抽象化:

```text
PERSON_SLOT_1 は TERM_SLOT_1 を 買った 。
```

ここで重要なのは、

- `は`
- `を`
- `買った`

という骨格であり、`新型量子加速器` の詳細意味は後で知識部に委ねればよい。

### 例2

原文:

`OpenAIは新しい埋め込みモデルを公開した。`

抽象化:

```text
ORG_SLOT_1 は TERM_SLOT_1 を 公開した 。
```

### 例3

原文:

`東京大学で超伝導量子干渉計を用いた。`

抽象化:

```text
ORG_SLOT_1 で TERM_SLOT_1 を 用いた 。
```

### 例4: 数式

元の式:

$$
\mathcal{L}_{\mathrm{meta}}(\theta, \phi) = \sum_{i=1}^{N} \alpha_i f_{\phi}(x_i)
$$

抽象化:

```text
FUNC_SLOT_1(VAR_SLOT_1, VAR_SLOT_2) = SUM(VAR_SLOT_3 = 1 .. CONST_SLOT_1, MUL(COEFF_SLOT_1[VAR_SLOT_3], FUNC_SLOT_2(VAR_SLOT_2, VAR_SLOT_4[VAR_SLOT_3])))
```

ここで重要なのは、

- `=`
- `SUM`
- `MUL`
- 関数適用
- 添字構造

という骨格であり、各記号名の詳細意味は後で知識部に委ねればよい。

### 例5: 正規表現

元のパターン:

$$
\texttt{[A-Z]\{3\}[0-9]\{4\}}
$$

抽象化:

```text
PATTERN_SLOT_1 = CONCAT(REPEAT(CLASS_UPPER, 3), REPEAT(CLASS_DIGIT, 4))
```

ここで重要なのは、

- `CONCAT`
- `REPEAT`
- `CLASS_UPPER`
- `CLASS_DIGIT`

という構造であり、個々のドメイン名は後で差し替え可能である。

## 抽象化の利点

### 1. 語彙爆発を抑えられる

大量の内容語や記号名を最初から全部扱わなくてよい。

### 2. 小さいモデルで回しやすい

固定語彙が構造中心になるため、表現部の学習対象がかなり減る。

### 3. 未知語に強くなる

未知語や未知記号が来ても破綻せず、まず構造骨格だけを理解できる。

### 4. 知識部との分業がしやすい

内容語の解像度を後段の知識参照に委譲できる。

### 5. 言語横断性が上がる

文法マーカーや構造演算子中心の設計は、他言語や他記法への拡張にも有利である。

## 抽象化の弱点

### 1. 意味の細部を落としすぎる危険

内容語や記号名そのものが本質的なタスクでは、抽象化しすぎると性能が落ちる。

### 2. 型付けを間違えると誤解釈が広がる

`ORG_SLOT` と `PERSON_SLOT`、`FUNC_SLOT` と `CONST_SLOT` の誤りなどは、後続処理に響く。

### 3. 固有名詞の同一性解決が必要

`東京大学` と `東大`、`\alpha_i` と別表記の係数記号を同一スロットにするかどうかは別途管理がいる。

## 共通IRとの接続

抽象化された語は、共通IRでは `entity`, `quantity`, `relation`, `choice`, `ref` などに接続される。

例:

```lisp
(stmt
  (assert
    (event
      (pred sym:buy)
      (arg
        (role s:"agent")
        (entity (id PERSON_SLOT_1) (surface s:"太郎")))
      (arg
        (role s:"theme")
        (entity (id TERM_SLOT_1) (surface s:"新型量子加速器"))))))
```

ここで `PERSON_SLOT_1` や `TERM_SLOT_1` は、必要に応じて後段で知識部に接続される。

数式や正規表現でも同様に、抽象化された要素は `quantity`, `formula`, `pattern`, `ref` などに接続される。

例:

```lisp
(stmt
  (define
    (formula
      (eq
        (apply FUNC_SLOT_1 VAR_SLOT_1 VAR_SLOT_2)
        (sum
          (value VAR_SLOT_3)
          (mul
            (index COEFF_SLOT_1 VAR_SLOT_3)
            (apply FUNC_SLOT_2 VAR_SLOT_2 (index VAR_SLOT_4 VAR_SLOT_3))))))))
```

## 推奨ポリシー

初期プロトタイプでは次を採用するのがよい。

1. 文法語、数式演算子、regex 構造演算子は固定語彙
2. 数量、単位、括弧、比較記号は固定語彙
3. 固有名詞、専門語、数式記号名、パターン名は型付きスロットに抽象化
4. 一般内容語や分野依存記号も低頻度なら抽象化
5. 未解決なら `choice` や `unknown` を許容
6. 詳細意味は後段の知識部で解決

## 一文での定義

この語彙抽象化ポリシーは、

`自然言語・数式・正規表現に共通して、構造骨格を作る高頻度要素だけを固定的に扱い、それ以外の内容依存要素は型付き匿名スロットへ抽象化することで、小さいモデルでも頑健に表現構造を扱えるようにする方針`

である。
